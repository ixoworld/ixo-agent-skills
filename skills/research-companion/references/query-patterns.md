# Query Patterns for Common Research Shapes

Search engines reward specificity and intent. Use these patterns as starting points and adjust to the topic.

## "Is this claim true?"

Search the claim and its negation, separately. Truth-seeking is symmetric.

```
"[exact claim]"
"[claim] debunked"
"[claim] evidence"
"[claim] is wrong"
site:[primary domain if known] [claim]
```

Cross-check across at least three independent sources before assigning a verdict.

## "What does the latest research say about X?"

Date-bound the query and prefer primary or aggregator domains.

```
[topic] systematic review 2025..2026
[topic] meta-analysis recent
site:scholar.google.com [topic]
site:arxiv.org [topic]
site:nih.gov [topic] OR site:who.int [topic]
"latest research" [topic] -opinion -blog
```

If the field is fast-moving, restrict to the last year. If slow-moving (e.g., classical physics), no date filter needed.

## "Compare A vs B"

Search both with and without comparison framing - vendor comparisons are often biased.

```
A vs B comparison
A vs B [neutral comparison source domain]
A pricing
B pricing
A limitations
B limitations
"switched from A to B" OR "switched from B to A"
A B benchmark
```

Find at least one source that has no commercial interest in either.

## "Brief me on X before my meeting"

Optimise for orientation, not depth. Hit the encyclopedic source first, then a recent reputable secondary, then look for one primary anchor.

```
[topic] overview
[topic] explained
[topic] [current year]
site:[reputable secondary] [topic]
[topic] criticism OR [topic] controversy
```

The controversy query surfaces what the orientation pieces leave out.

## "Market scan / competitor scan"

```
[market] vendors
[market] competitive landscape
[market] market share [recent year]
"alternatives to [vendor]"
"vs [vendor]" reviews
"[vendor]" review
"[vendor]" complaint OR "[vendor]" issue
```

For competitor scans, search the competitor's own site for their stated positioning, then search third-party reviews for how customers describe them.

## "What are the current best practices for X?"

Best-practice content is often stale. Date-bound aggressively.

```
[topic] best practices 2026
[topic] guide [current year]
[topic] [authoritative org] guidance
"how to [topic]" site:[reputable secondary]
"common mistakes" [topic]
```

The "common mistakes" angle often surfaces practical truth that best-practice posts smooth over.

## "Who is [person]?"

```
"[full name]" "[employer or known affiliation]"
site:linkedin.com "[name]"
"[name]" interview
"[name]" [recent year]
```

Disambiguate by pairing the name with a known fact (employer, location, field) - common names produce noise otherwise.

## Patterns that waste time

- Generic queries like `[topic]` without qualifiers - too noisy.
- Queries with `best` and no context - SEO spam.
- Asking the search engine a full natural-language question on contested topics - it surfaces blogspam first.
- Re-searching the same engine with rephrased queries instead of trying a different angle.

## When the web is the wrong tool

For some research, the web is not the right substrate:

- **Specific paywalled academic literature**: use a library or academic database.
- **Recent regulatory filings**: go to the regulator's site directly.
- **Real-time prices, schedules, weather**: use the authoritative source's live page.
- **Internal company facts**: ask the user; the web won't have them.

State the limitation. Do not pretend the web answered a question it didn't.
